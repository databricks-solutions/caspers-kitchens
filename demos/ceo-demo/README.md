# CEO Dashboard Demo Flow

A live executive intelligence demo built on Casper's Kitchens — a ghost kitchen network running 16 restaurant brands across 8 locations (US: San Francisco, Silicon Valley, Bellevue, Chicago; EMEA: London, Munich, Amsterdam, Vianen).

The CEO Dashboard is a Databricks App with a full-stack experience: revenue charts, a live location health map, and a chat interface backed by a 7-agent Multi-Agent Supervisor. The demo story runs in three parts.

---

## Deploy

```bash
databricks bundle deploy -t ceo --var "catalog=<your_catalog>"
databricks bundle run caspers -t ceo --params "CATALOG=<your_catalog>"
```

The job runs these stages in sequence:

```
Canonical_Data → Lakeflow_Pipeline
             → Inspection_Data → Inspection_Knowledge_Agent
             → CEO_Document_Data → CEO_Knowledge_Agents
             → CEO_Revenue_Genie + CEO_Ops_Genie
             → CEO_Supervisor
             → CEO_Lakebase
             → CEO_App
```

## Part 1 — Lakebase: Session-Aware Executive Chat

**The point:** The CEO asked us to build an AI assistant — but a stateless chatbot isn't good enough at the C-suite level. The CEO needs to pick up conversations where they left off, across devices and sessions, without losing context. That requires a real database. The hard part is doing it inside Databricks, without spinning up a separate cloud database, managing secrets, or leaving the governance boundary. Lakebase solves this: managed PostgreSQL, inside Databricks, governed by Unity Catalog.

### What to show

1. Open the app and start a new session
2. Close the tab and reopen — session persists
3. Show the data live in Lakebase with SQL
4. Demonstrate PITR and branching
5. Explain the credential and connection architecture

---

**1. Open the CEO Dashboard app.**

The left sidebar shows session history. Click **+ New Session** and ask:

> "Which location has the highest revenue this week?"

The Revenue Genie runs a single SQL query — response comes back in seconds. Notice the session appears in the sidebar with an auto-generated title — the title is generated by the AI from the first message, not typed by the user.

**2. Demonstrate persistence.**

Close the tab entirely and reopen the app. The session is still in the sidebar with its title intact. Click it — all messages reload instantly from Lakebase. This is not browser cache. This is a database.

**3. Show the data in Unity Catalog.**

Open **+ Lakebase** and navigate to **+ Tables**. Run a quick SQL query:

```sql
SELECT s.title, m.role, LEFT(m.content, 120) AS preview, m.created_at
FROM ceo_sessions s
JOIN ceo_messages m ON s.id = m.session_id
ORDER BY m.created_at DESC
LIMIT 10;
```

The rows you just created appear immediately. These are standard PostgreSQL tables — fully queryable, auditable, and governed by Unity Catalog permissions.

**4. Demonstrate enterprise Lakebase features.**

These are the features that make Lakebase production-ready, not just a convenient database.

**Point-in-time recovery:**
In the Databricks UI, navigate to **Compute → Lakebase → `<catalog>-ceo-sessions`**. Show the **Restore** option. Explain: if a bad deployment truncated the sessions table, you can restore the database to any point in the last 7 days with one click — no backups to manage, no restore scripts to write. This is the same guarantee you'd expect from a managed cloud database, but it's included automatically.

**Branching:**
Show the **Branches** tab on the Lakebase project. Explain: you can create an isolated branch of the full database in seconds — it shares storage with the main branch until writes diverge (copy-on-write). This means a developer can test a schema migration against real production data without touching production. When they're done, the branch is deleted. No cloning, no manual data copying.

> **Demo tip:** If time allows, create a branch live: click **New Branch**, name it `demo-test`, run a destructive query against it, then show that the main branch is untouched.

**5. Explain the architecture.**

The app connects to Lakebase using short-lived credentials generated by the Databricks SDK (`w.postgres.generate_database_credential()`). A background thread refreshes the token every 48 minutes so connections never drop during a long session. The app never stores a password — authentication is identity-based, issued by Databricks, and the connection always uses `sslmode=require`.

There is no external cloud database. No RDS, no Cloud SQL, no secrets manager, no VPN. The database lives inside Databricks, under the same IAM boundary as the rest of the data.

### Key talking points
- Lakebase is managed PostgreSQL that lives inside Databricks — no separate cloud database, no secrets management, no VPN
- Credentials are identity-based: the app authenticates as a service principal, the token is issued by Databricks, and the connection uses `sslmode=require`
- Unity Catalog governs the tables — you can write SQL over them, set row-level security, and audit access
- The token refresh pattern means the app stays connected indefinitely without manual intervention

## Part 2 — Document Processing: 5 Knowledge Assistants + 2 Genie Spaces

**The point:** So you thought the memory was the hard part? No — it's the brain. The CEO's knowledge lives across hundreds of PDFs in different formats, styles, and structures: inspection reports, legal filings, audit findings, regulatory permits, consulting decks. You can't write a parser for each one. Instead: Genie for structured data, Knowledge Assistants for documents, and a Multi-Agent Supervisor that routes any question to the right agent automatically.

### What to show

1. A revenue question routing to Revenue Genie (live time-range SQL)
2. A food safety question routing to Operations Genie (tabular scores across all locations)
3. A document drill-down routing to Inspection KA (from score → actual PDF)
4. A legal exposure question routing to Legal KA (PDFs in UC Volume → structured answer)
5. The board summary question hitting all 7 agents at once

**1.** `"Which location had the highest revenue this month? How does that compare to last month?"`

Routes to Revenue Genie. Runs a time-range comparison against `gold_location_sales_hourly` — live SQL, no pre-built report. Returns a ranked table with revenue figures and month-over-month delta. Point out that the Genie wrote and executed the SQL itself.

**2.** `"What is the current food safety grade for each location? Which ones are failing?"`

Routes to Operations Genie. Queries `food_safety.inspections` and returns a table with scores and grades across all 8 locations. Chicago will show grade F (score 69) — its marker on the map should turn red. This is the natural setup for the next question.

**3.** `"What happened during the Chicago food safety inspection in January? Were there critical violations?"`

Routes to Inspection KA. Returns details from the actual PDF: score 69 (grade F), critical violation V-103 ("Employee handling food without gloves"), 3-day corrective deadline. Click **Document Viewer** in the sidebar — the referenced inspection report appears automatically. This is the moment that shows structured data → document drill-down.

**4.** Open the `legal_complaints` volume in Catalog Explorer. Show the PDFs. Then ask:

> `"Do we have any active high-risk legal cases? What's the total financial exposure?"`

Routes to Legal KA. Returns case numbers (CK-XX-XXXX), risk levels (HIGH/MEDIUM/LOW), and amounts at stake — all extracted from those same PDFs the audience just saw. The **Hire Consultants** or **Fire Consultants** button surfaces based on the response.

**5.** Open the CEO Supervisor tile. Show the `description`, `instructions`, and `examples` fields. Explain that this is where routing behavior is defined — the supervisor reads these to decide which agent to call for each question.

**6.** `"Give me a one-page board deck summary: revenue performance, top operational risk, legal exposure, and one strategic recommendation."`

All 7 agents fire in parallel. The supervisor synthesizes their responses into a single executive answer. This is the money question — the CEO no longer needs 7 dashboards, 7 reports, and 7 calls.

### Key talking points
- Genie handles structured data with live SQL — no stale dashboards, no pre-built queries
- KAs handle unstructured documents — PDFs in UC Volumes become queryable knowledge the moment they're uploaded
- The supervisor routes on intent, not keywords — "the Chicago situation" triggers both Operations Genie and Inspection KA without any hardcoded rules
- The Document Viewer auto-populates from IDs extracted from the response — it's not a static file browser, it follows the conversation

## Part 3 — MLflow: Tracing and Evaluating the Supervisor

**The point:** So you thought processing documents was the hard part? The real challenge is making your users *trust* an AI system. A CEO won't act on a recommendation they can't explain to the board. MLflow gives you two things: full observability (every question generates a trace showing which sub-agent was called, what it returned, and how long it took) and systematic quality measurement (LLM-as-judge evaluation with built-in and custom scorers). Together they turn the CEO Dashboard from a demo into a production system you can stand behind.

### What to show

1. Open the production traces experiment from the right sidebar of the app
2. Walk through a trace — routing decisions and latency breakdown
3. Open the evaluation notebook — dataset design and scorer logic
4. Show pre-run evaluation results — per-question scores across all 4 judges
5. Show the trace-derived dataset and the production→eval flywheel
6. Every AI decision is now auditable

### Important: add LLM judges and datasets via UI, no automatic way as of now 

---

> **Two MLflow experiments, one story.** The sidebar **Supervisor Traces ↗** link opens `/Serving/{endpoint}` — the experiment Databricks auto-creates to capture every live request. The evaluation notebook writes its runs to a *separate* experiment: `/Users/{you}/ceo-supervisor-eval`. Keep both tabs open during the demo.

---

**1. Open the production traces experiment.**

Click **Supervisor Traces ↗** in the right sidebar of the CEO Dashboard (under the MLflow section). This is the `/Serving/{endpoint}` experiment — Databricks writes here automatically whenever a request hits the supervisor, no instrumentation code required.

> **Demo tip:** Run at least one Part 2 question before switching to MLflow. Traces appear within seconds of a response completing.

**2. Pick the board deck summary trace.**

Find the trace for `"Give me a board deck summary: revenue performance, top operational risk, legal exposure, and one strategic recommendation."` Expand it:

- The **root span** is the MAS supervisor call — shows the full user question and final synthesized response
- **Child spans** show each sub-agent invocation: `revenue-analytics`, `operations-intelligence`, `inspection-reports`, `legal-complaints`, `regulatory-compliance`, `audit-findings`, `consultancy-strategy`
- Each span has: the routed question sent downstream, the sub-agent's raw response, latency, and token counts

**3. Show the routing decision.**

Point out which agents fired and which didn't. For the board summary, all 7 spans should be active. For a pure revenue question like `"Which location has the highest cancellation rate?"`, only the `revenue-analytics` span fires — the KA agents are idle.

This is the supervisor doing intelligent routing, not keyword matching. Ask `"the Chicago situation"` and it calls both `operations-intelligence` (food safety score, complaints) and `inspection-reports` (the actual inspection PDF) in parallel.

**4. Show the latency breakdown.**

Point to the waterfall. Genie queries (live SQL over Delta) are typically the bottleneck at 10–30s on a cold warehouse. KA document retrieval runs in 2–8s. The supervisor synthesis adds 3–6s on top. This is where you can see whether pre-warming the warehouse or adding a caching layer would have the most impact.

**5. Open the evaluation notebook.**

Click **Evaluation Notebook ↗** in the right sidebar (or navigate to `demos/ceo-demo/ceo_evaluation` in the workspace). This notebook creates its own experiment: `/Users/{you}/ceo-supervisor-eval`. Walk through the seven sections:

**Evaluation dataset — 10 questions across 7 agent domains.**

Each row has `inputs` (the question) and `expectations` with two fields:

| Field | Purpose |
|---|---|
| `guidelines` | Per-question pass/fail criteria for the `ExpectationsGuidelines` scorer |
| `expected_agent` | Domain label (`revenue`, `legal`, `audits`, …) for the `routing_accuracy` scorer |

Guidelines are behavioral — they test *how* the agent responds, not the current data values. `"Must cite a case number in CK-XX-XXXX format"` survives a simulator update. `"Must say Chicago has 11.7% cancellation rate"` breaks the moment the data shifts. That's why there is no `Correctness` scorer.

**Four scorers — 2 built-in, 2 custom:**

| Scorer | Type | What it checks |
|---|---|---|
| `ExpectationsGuidelines` | Built-in LLM judge | Per-row pass/fail against each question's `guidelines` list |
| `Safety` | Built-in LLM judge | Harmful or inappropriate content in responses |
| `routing_accuracy` | Custom `@scorer` | Does the response vocabulary match the expected sub-agent domain? |
| `cites_specific_data` | Custom `@scorer` | Does the answer contain numbers, dates, case IDs — or does it hedge? |

`routing_accuracy` uses domain vocabulary signals (`"CK-"`, `"risk level"`, `"counsel"` for legal; `"audit"`, `"KPMG"`, `"critical finding"` for audits) to infer whether the MAS routed correctly — without needing access to internal trace data.

`cites_specific_data` detects concrete patterns — percentages, dollar amounts, ISO dates, inspection scores, ROI multiples — and penalizes hedges like `"I cannot access"` or `"please consult your team"`.

**6. Show the evaluation results.**

Open the `/Users/{you}/ceo-supervisor-eval` experiment and click the `ceo-supervisor-full-eval` run. Show the per-question scorecard — which questions passed all 4 scorers, which didn't. Use the failures as a teaching moment:

| Score | What it means | Fix |
|---|---|---|
| `routing_accuracy` < 0.5 | MAS called the wrong sub-agent | Improve the `description` field for that sub-agent in `ceo_supervisor.ipynb` |
| `cites_specific_data` = 0.0 | Hedged/empty response | Sub-agent retrieval failed — check endpoint status and UC Volume contents |
| `guidelines` < 0.5 | Answer doesn't satisfy the stated criteria | Improve the sub-agent's `instructions` in `ceo_knowledge_agents.ipynb` |
| `safety` < 1.0 | Potentially harmful content | Review the full response — rare but warrants investigation |

**7. Show the trace-derived dataset — the production→eval flywheel.**

This is the part that makes the architecture feel production-grade. The notebook fetches the last 50 successful traces from the `/Serving/{endpoint}` experiment, extracts real CEO questions from the root span inputs, and builds a second evaluation dataset from them:

```
Production traffic → MLflow traces → filter interesting traces → eval dataset → ceo-supervisor-trace-eval run
```

Show the two runs side by side in the MLflow Experiments UI:

| Run | Dataset | What it tells you |
|---|---|---|
| `ceo-supervisor-full-eval` | 10 curated questions with behavioral guidelines | How well the supervisor handles the demo script |
| `ceo-supervisor-trace-eval` | Real CEO questions from production traces | General quality over real traffic |

The trace-derived run uses `Guidelines()` (a single global guideline applied to every row) instead of `ExpectationsGuidelines()`, since real traffic doesn't come with per-question criteria. The `cites_specific_data` and `routing_accuracy` scorers work the same way either way.

**8. Tie back to the CEO story.**

> "If the board asks why the AI recommended restructuring the Chicago operation — this trace is the answer. Every routing decision, every document retrieved, every synthesis step is recorded. This is AI you can audit, not AI you have to trust blindly."

### Key talking points
- Traces are automatic — the Databricks MAS platform writes them; no `mlflow.trace()` calls anywhere in the app code
- Two experiments serve different purposes: `/Serving/...` for production observability, `/Users/.../ceo-supervisor-eval` for quality measurement
- The eval dataset tests agent *behavior*, not current data values — it stays valid across simulator updates and new locations
- `routing_accuracy` and `cites_specific_data` are custom scorers written in ~15 lines each using `@scorer` — no framework boilerplate
- The trace-derived dataset closes the production→eval loop: every real CEO question can become a future test case
- Next steps: **MemAlign** to calibrate the `Guidelines` judge to your specific definition of a good CEO response; `mlflow.genai.optimize_prompts()` to automatically improve supervisor instructions based on failing scores

> **Pre-demo setup:** Run `ceo_evaluation.ipynb` cells 1–5 before the live demo (takes 5–15 min, makes 10 calls to the supervisor). Open the `/Users/{you}/ceo-supervisor-eval` experiment in a second tab — results will be ready when you reach Part 3.

```
apps/ceo-dashboard/
├── app.yaml                    # Lakebase + env config (written by ceo_lakebase stage)
├── requirements.txt
├── index.html                  # SPA: left sidebar, revenue charts, map, chat, doc viewer
└── app/
    ├── main.py                 # FastAPI: sessions, chat streaming, revenue, locations

stages/
├── ceo_document_data.ipynb     # Runs 4 generators, uploads to UC Volumes
├── ceo_knowledge_agents.ipynb  # 4 new KAs (legal, regulatory, audits, consultancy)
├── ceo_genie_spaces.ipynb      # Revenue Genie + Operations Genie
```
